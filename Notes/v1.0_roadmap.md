# V1.0 Completion Roadmap (Pre-BidAsk)

Before integrating BidAsk data (V2.0), we should maximize the potential of `Tick` data and harden the infrastructure.

## 1. Advanced Tick-Based Indicators (Missing Alpha)

Currently, we only have VWAP and basic Momentum. We are missing **Volatility** and **Statistical** indicators which are crucial for "Mean Reversion" strategies.

### A. Volatility & Stat Indicators
*   **Bollinger Bands / Keltner Channels**:
    *   Need `Standard Deviation` (std) calculation.
    *   Use: Identifying "Squeeze" (low vol) before "Breakout" (high vol).
*   **Z-Score (Standard Score)**:
    *   Quantifies how "rare" a price move is.
    *   Use: Mean reversion signals when Z-Score > 2 or < -2.

### B. Momentum Oscillators (Fast)
*   **RSI (Relative Strength Index)**:
    *   Standard but effective. Can be optimized to O(1) or O(N) in Numba.
*   **ROC (Rate of Change)**:
    *   Pure velocity of price.

### C. Volume Profile (Session Logic)
*   **POC (Point of Control)**: The price level with the highest traded volume in the current session.
*   **VAH/VAL (Value Area High/Low)**: Where 70% of volume occurred.
*   *Why V1.0?* It can be built purely from Ticks (Price/Vol) without Order Book. It provides the strongest Support/Resistance levels.

---

## 2. Infrastructure Upgrade: Shared Memory (Critical)

Current bottleneck: Python GIL (Global Interpreter Lock). Even with `nogil=True` in Numba, the Python loop in `CoreProcessor` still holds the lock, fighting with the Dashboard server.

### The Solution: `SharedMemoryRingBuffer`
Replace current `np.zeros` array with `multiprocessing.shared_memory.SharedMemory`.

#### Architecture Change
*   **Process A (Ingestion)**: Just writes to Shared Memory. Zero calculation. Infinite throughput.
*   **Process B (Strategy/Dash)**: Read-Only from Shared Memory. Calculates indicators. Runs Dash.
*   **Benefit**: If Dash crashes or calculation hangs, data recording **never stops**.

---

## 3. The "Time Machine" (Replay Engine)

We have `consume_history`, but we need a robust **Event-Driven Backtester**.

*   **Goal**: Run the *exact same* strategy code on historical parquet files.
*   **Why**: You cannot optimize parameter (e.g., SMA period, Thresholds) without this.

1.  **[High Priority] Shared Memory RingBuffer**:

2.  **[Medium Priority] Volatility Indicators (Bollinger/Z-Score)**:
    *   Implement `calc_std` and `calc_zscore` in `numba_engine.py`.
    *   Add to Dashboard.

3.  **[Medium Priority] Volume Profile POC**:
    *   Implement a fast Hash Map or Array-based Volume Profile in Numba.

4.  **[Low Priority] RSI/ROC**:
    *   Nice to have, but less critical than Volatility.
